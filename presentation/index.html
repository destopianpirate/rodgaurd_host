Problem Statement

The Engineering Problem

  • Indian roads have ~33 pothole-related fatalities per day (MoRTH, 2023)
  • Manual road inspection is slow, expensive, and reactive
  • No low-cost, real-time system exists for continuous, automated pothole mapping from moving vehicles
Objective: Design an edge-deployed, vehicle-mounted system that detects potholes in real-time using computer vision, geo-tags them via GPS, alerts the driver, and streams data to a live web dashboard — all under ₹8,000 hardware cost.

Technical Approach — AI Model

YOLOv8 Nano Architecture

  • Model: YOLOv8n — 3.2M parameters, single-class (pothole)
  • Input: 640×480 BGR frames from Picamera2
  • Confidence threshold: 0.50
  • Severity classification by bounding-box area ratio:
SeverityArea Ratio
Critical≥ 15% of frame
High≥ 8%
Medium≥ 3%
Low< 3%

Design Trade-off

Chose YOLOv8n (nano) over YOLOv8s/m for 2× faster inference on ARM CPU, accepting ~3% lower mAP. Edge speed > cloud accuracy.

3-Thread Architecture

Main Thread
Camera capture + OpenCV display
~30 FPS smooth rendering
YOLO Thread
AI inference on latest frame
2s cooldown between detections
GPS Thread
Continuous NMEA parsing
Non-blocking position reads

Threads share state via Python GIL. YOLO thread copies frame before processing to avoid blocking the display loop.

Resolution Logic

When a vehicle drives past a known pothole but the camera sees no pothole:

  1. Increment absent_count
  2. If absent_count ≥ threshold → mark as resolved
  3. Resolved potholes disappear from the dashboard

Haversine Distance Formula

d = 2R · arctan2(√a, √(1−a)) where: a = sin²(Δφ/2) + cos(φ₁)·cos(φ₂)·sin²(Δλ/2) R = 6,371,000 m (Earth radius) φ = latitude, λ = longitude Threshold: d < 30 m → same pothole

Why 30m Radius?

GPS accuracy of NEO-M8N is ±2.5m (ideal). At 30 km/h city speed with 2s detection cooldown, vehicle travels ~17m per cycle. 30m covers positional uncertainty + vehicle displacement.

Challenges Encountered During Development

🔵

Blue Camera Tint

Problem: Picamera2 captured RGB; OpenCV expects BGR. Double color-swap produced blue-tinted frames.

Fix: Changed capture format to BGR888, eliminated redundant cvtColor() call.

🔔

Buzzer Won't Stop

Problem: Active buzzer module uses active-low logic. Setting GPIO HIGH = buzzing (opposite of expected).

Fix: Switched to INPUT/OUTPUT mode toggling. INPUT = high-impedance = silent. OUTPUT LOW = buzzing.

🌡️

Thermal Throttling

Problem: Pi 4 throttles CPU at 80°C during sustained YOLO inference. Detection speed dropped from 8→3 FPS.

Fix: Added heatsinks + Pi case with passive ventilation. Implemented 2s cooldown to reduce sustained load.

📍

GPS "No Fix" Indoors

Problem: NEO-M8N requires clear sky view. 30-90s cold start. No fix indoors = no geo-tagged detections.

Fix: Added fallback: detections without GPS fix are logged locally but not sent to server. GPS status shown in UI overlay.

📂

Wrong Model Path on Pi

Problem: Config pointed to training output path (../runs/detect/.../best.pt) which doesn't exist on Pi.

Fix: Reverted to relative best.pt in same directory. Model load failure caused zero detections (silent failure).

📶

WiFi Drops in Vehicle

Problem: Moving vehicle exits WiFi range. HTTP requests fail, detections lost.

Fix: Implemented offline queue (offline_queue.json). Auto-flushes when connectivity restores. Max 1,000 buffered entries.

Limitations & Future Work

Current Limitations

LimitationImpact
No nighttime capabilityCamera unusable without headlight illumination
Single-class detectionOnly potholes; no cracks, speed bumps, or debris
No pothole depth estimation2D camera cannot measure depth; severity is only by area
WiFi-dependent connectivityNo cellular fallback for rural/highway use
8 FPS on bare Pi CPUMay miss potholes at highway speeds (>60 km/h)
No multi-modal confirmationCamera and IMU are independent; false positives from shadows/leaves
Single-vehicle testedCollaborative verification not tested at fleet scale

Proposed Improvements

  • Night Vision: CLAHE pre-processing or IR camera for nighttime detection
  • Multi-Modal Fusion: Only report pothole if camera AND IMU both confirm — reduce false positives
  • Coral TPU Integration: $25 accelerator → 25 FPS, enabling highway deployment
  • 4G/LTE Module: Replace WiFi with cellular for continuous rural connectivity
  • LiDAR Depth Sensing: 1D LiDAR (Garmin Lidar-Lite) for actual pothole depth measurement
  • Comfort Routing: Route planning that minimizes exposure to pothole-dense zones
  • Voice Alerts: TTS engine for context-aware warnings ("Critical pothole in 50m")
  • Automated Municipal Reports: PDF generation with GPS, severity, and timestamps

Key Takeaway

RoadGuard demonstrates that real-time, edge-deployed pothole detection is feasible on a sub-₹8,000 Raspberry Pi platform. The critical engineering contribution was solving the lab-to-road accuracy gap through targeted data augmentation — recovering from a 42.5% field accuracy to 88% — making the system viable for deployment on Indian roads.
